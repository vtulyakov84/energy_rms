Направление проекта: **Разработка системы управления заявками.**

### Обоснование использования микросервисной архитектуры:
В связи с неизбежной "гибелью" монолитных архитектур построения программных решений, которые пользовались спросом с 1990-х годов, соавтором проекта, при реализации определяется неизбежность перехода в облако и смерти десктопных монолитов, в связи с **эволюционной сложностью**, **сложностью масштабирования** и **естественным отбором в сфере IT-систем**.

Десктопный монолит с удалённой БД обречён, потому что он неспособен эволюционировать без разрушительных последствий, любое изменение которого требует переустановки у всех пользователей, исправление бага в одном модуле рискует сломать всё приложение, а масштабирование упирается в железо одного компьютера.

Микросервисы, в частности при использовании языка программирования `Golang` с базой данных `PostgreSQL` в в среде виртуализации распределенных процессов`Docker`, соавтором проекта рассматривается как  архитектура, в которой каждый компонент живёт "своей жизнью". Один сервис можно переписать, обновить или увеличить его мощность, не трогая остальные. Контейнеры гарантируют изоляцию зависимостей, а горутины Go позволяют утилизировать облачные ресурсы на 100%, "платя" только за реальную нагрузку.

Поэтому эволюция неизбежно приведёт всё в облако. Десктопные монолиты исчезнут не потому, что они «плохие», а потому что бизнесу станет экономически невыгодно их поддерживать. Поддержка сотен версий ОС, ручное обновление, невозможность геораспределения и постоянные утечки строк подключения к БД сделают их вымирающим видом, как мэйнфреймы, COBOL, дискеты 5.25", медная проволока на спицы велосипеда и ламповые телевизоры.

## Ошибки, поквлиявшие на что-либо в проекте

#### Ошибка 1, оказала влияние на структуру проекта
При нахождении каталога `postgresql/data` в составе проекта golang,
из-за отсутствия доступа к этой папке, команда `go mod tidy` выдавала
ошибку доступа к каталогу - `permission denide`, что не позволяло на выходе команды получать файл `go.sum`, с вытекающими неудобствами,
поэтому пришлось проект сервиса API-шлюза вынести в отдельный каталог `./gatewayapi`.

##### Ошибка 2, оказала влияние на место хранения (вне структуры каталогов проекта) файлов базы данных PostgreSQL.
Из-за отсутствия прав на папку `postgresql/data` при работе с командами `git...`, пока решено размещать файлы базы даннх вне проекта, по пути `/var/lib/postgresql/data`. Внимание!, при расположении файлов БД по указанному пути, убедитесь, что каталог остуствует.
Создайте папку по указанному пути и выполните `$sudo docker compose up -d`, с указанием пути в файле `docker-compose.yml`.
Если такой каталог, перед запуском `$ sudo docker compose up -d`, отсутствует!, создайте его `$ sudo mkdir -p /var/lib/postgresql/data`, указав этот каталог также в `docker-compose.yml` в секции расположения файлов БД (см файл repo `./docker-compose.yml`).
